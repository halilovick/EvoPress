# Compression settings

## Depth pruning baseline

Example:

```text
drop sparsity = 0.125 or 0.25
```

For TinyLlama, if it has 22 layers:

```text
0.125 -> about 2 blocks/subblocks
0.25  -> about 5 blocks/subblocks
```

Start small:

```bash
SPARSITY=0.125
```

---

## Quantization baseline

Use only a restricted module set first:

```text
q_proj only
```

Then later:

```text
q_proj, k_proj, v_proj, o_proj
```

Bitwidths:

```text
2, 3, 4
```

Target/calibration bitwidth:

```text
3-bit
```

---

## Joint search

First version:

```text
depth pruning + q_proj quantization
```

Candidate state:

```python
candidate = {
    "drop": {
        "attn": [...],
        "mlp": [...]
    },
    "quant": [3, 3, 4, 2, ...]
}
```

Where:

- `drop` controls skipped attention/MLP modules
- `quant` controls bitwidth level for each quantized layer

# Implementation plan

Do it in four stages.

## Stage 1 — confirm depth pruning on TinyLlama

Run `evo_drop_search.py` with TinyLlama and small settings.

This checks:

- model loads
- `get_layers()` works
- `get_attn_layer_name()` works
- `get_mlp_layer_name()` works
- dummy forward patch works

## Stage 2 — generate tiny quant database

Run `quant.py` with:

```bash
MODEL="TinyLlama/TinyLlama-1.1B-Chat-v1.0"

```

Output:

```text
outputs/quant_db_tiny/TinyLlama-1.1B-Chat-v1.0/3bit/<layer_name>/<bit>.pth
```

---

## Stage 3 — run quant-only search

Run `evo_quant_search.py` on that small database.

This gives quant-only baseline.

---

## Stage 4 — implement `evo_joint_search.py`

Start from `evo_drop_search.py`, then add:

- `load_quant_layers()` from `evo_quant_search.py`
- joint candidate state
- joint mutation
- joint apply function

Minimal version:

```python
def apply_joint_state(model, layers, layer_names, candidate):
    load_drop_state(model, layers, candidate["drop"])
    load_quant_state(model, layer_names, candidate["quant"])
```

Mutation:

```text
with 50% probability:
    mutate drop state

with 50% probability:
    mutate quant state
```

At first, do not overcomplicate with a unified budget. Keep separate fixed budgets:

```text
drop budget fixed
quant average bitwidth fixed
```

This is not the final thesis version, but it is a clean first prototype.

# Important: first joint prototype should use separate budgets

For the first test, do **not** implement full total-bit budget yet.

Use:

```text
drop budget: fixed number of dropped blocks
quant budget: fixed average bitwidth
```

Why?

Because it is easier and closer to existing EvoPress mutation operators.

Then later thesis extension can improve this to:

```text
unified model-size budget
```

This gives you a natural progression:

```text
Prototype 1:
Joint search with separate budgets

Prototype 2:
Joint search with unified effective-size budget
```

# Expected result

Will it definitely beat both separate searches?

Not guaranteed, especially with tiny settings.

But even if it does not, the experiment is still useful because it answers:

```text
Can the code support joint states?
Can we apply two compression methods at once?
Does the search run?
What are the practical bottlenecks?
```

For the thesis, that is already valuable.

With enough generations/tokens, I would expect joint search to sometimes outperform one method alone at similar total compression, because it can trade off:

```text
drop less important subblocks
while preserving/allocating better bitwidths elsewhere
```

# Recommended next step

First, test TinyLlama depth pruning.

Create:

```bash
scripts/run_drop_search_tiny.sh
```

with:

```bash
MODEL="TinyLlama/TinyLlama-1.1B-Chat-v1.0"
CALIB_DATA="wikitext2"

SEQUENCE_LENGTH=256
CALIB_TOKENS=2048
SPARSITY=0.125
GENERATIONS=3

CONFIG_PATH="./outputs/drop_tiny_test"

mkdir -p "$CONFIG_PATH"

python evo_drop_search.py \
    --model_name_or_path "$MODEL" \
    --sparsity "$SPARSITY" \
    --calibration_data "$CALIB_DATA" \
    --calibration_tokens "$CALIB_TOKENS" \
    --calibration_sequence_length "$SEQUENCE_LENGTH" \
    --eval_every 1 \
    --eval_datasets wikitext2 \
    --eval_tokens 8192 \
    --eval_sequence_length "$SEQUENCE_LENGTH" \
    --population_size 1 \
    --generations "$GENERATIONS" \
    --offspring 4 \
    --initially_generated 8 \
    --initial_tokens 256 \
    --survivors_per_selection 2 1 \
    --tokens_per_selection 256 1024 \
    --fitness_fn kl \
    --drop_config_dir "$CONFIG_PATH" \
    --dtype float16 \
    --attn_implementation sdpa
```

If this works, then we move to the tiny quant database.
